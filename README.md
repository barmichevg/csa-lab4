## Язык программирования

Реализован минимальный диалект Forth. Программа состоит из потока слов. Все вычисления выполняются через стек данных. Поддержаны процедуры, переменные, буферы, условные переходы, циклы `begin/until`, строки с префиксом длины, обработчик прерывания и `execution token`.

### Синтаксис

Синтаксис описан в расширенной форме Бэкуса-Наура.

```ebnf
program        = { top-item } ;

top-item       = definition
               | irq-definition
               | declaration
               | word ;

definition     = ":" , name , { word } , ";" ;
irq-definition = ":irq" , { word } , ";" ;

declaration    = "variable" , name
               | "buffer" , name , integer ;

word           = integer
               | name
               | "'" , name
               | "execute"
               | "if" | "else" | "then"
               | "begin" | "until"
               | "dup" | "drop" | "swap" | "over"
               | "+" | "-" | "*" | "/" | "mod"
               | "=" | "<" | ">"
               | "@" | "!"
               | "ei" | "di" | "iret" | "halt"
               | pstring
               | print-string ;

pstring        = "p\"" , { string-char } , "\"" ;
print-string   = ".\"" , { string-char } , "\"" ;
name           = letter , { letter | digit | "-" | "_" | "?" | "!" } ;
integer        = [ "-" ] , digit , { digit } ;
comment        = "\\" , { any-char-except-newline } , newline ;
```

### Семантика

#### Стратегия вычислений

Вычисления строгие и последовательные. Слова выполняются слева направо. Операнды передаются через стек данных: числовой литерал кладёт значение на стек, арифметическая операция снимает аргументы со стека и кладёт результат обратно.

#### Области видимости

Все имена глобальные. Процедуры, переменные и буферы находятся в одной области имён. Повторное объявление имени запрещено. Локальные переменные не поддерживаются.

#### Типизация

Типизация отсутствует. Все значения представлены 32-битными машинными словами. Одно и то же слово может интерпретироваться как число, адрес, символ или флаг. Для логических значений используется соглашение: `0` -- ложь, `1` -- истина.

#### Строки `pstr`

Строки хранятся в памяти данных в формате Pascal string: первая ячейка содержит длину строки, следующие ячейки -- коды символов. Один символ занимает одну 32-битную ячейку.

```text
p"Hi"

addr + 0 : 2
addr + 1 : 72   ('H')
addr + 2 : 105  ('i')
```

Слово `p"..."` размещает строку в памяти данных и кладёт её адрес на стек. Для вывода строки используется библиотечная процедура `type`. Слово `."..."` является синтаксическим сокращением: оно размещает Pascal-строку в памяти данных и сразу генерирует вызов `type`. 

#### Комментарии

Комментарий начинается с символа `\` и продолжается до конца строки.

## Организация памяти

Архитектура гарвардская: память команд и память данных разделены. Обе памяти адресуются по 32-битным словам. В бинарном файле одно слово хранится как 4 байта, но внутри модели адрес -- это номер слова, а не байтовый адрес.

### Память команд

```text
Program memory
+------------------------------+
| 0000 : jmp main              |
| 0001 : jmp irq_handler       |
| .... : stdlib procedures     |
| .... : user procedures       |
| .... : main                  |
+------------------------------+
```

Адрес `0` используется как reset vector. Адрес `1` используется как interrupt vector. Далее размещаются процедуры стандартной библиотеки, пользовательские процедуры и основной код программы.

### Память данных

Используется единое 16-битное адресное пространство из `2^16 = 65536` адресов. Адрес остаётся пословным: один адрес выбирает одно 32-битное слово. Верхние адреса зарезервированы под MMIO.

```text
Data address space
+-----------------------------------+
| 0000 : stdlib variables           |
| .... : input buffer               |
| .... : user variables/buffers     |
| .... : static pstrings            |
| FFEF : last regular memory word   |
| FFF0 : MMIO_IN_DATA               |
| FFF1 : MMIO_IN_STATUS             |
| FFF2 : MMIO_OUT_DATA              |
| FFF3 : MMIO_IRQ_ACK               |
| FFF4..FFFF : reserved/unmapped     |
+-----------------------------------+
```

Строковые литералы, переменные и буферы размещаются статически на этапе трансляции. Обычные адреса памяти данных проходят через cache. MMIO распознаётся декодером адреса до cache и никогда не кешируется.

### Стеки

Процессор использует два стека.

- `Data Stack` -- основной стек операндов. Через него выполняются арифметика, сравнения, работа с адресами и памятью.
- `Return Stack` -- стек адресов возврата. Он используется командами `call`, `ret`, `execute`, входом в прерывание и `iret`. 

Архитектурно верхушка стека обозначается как `TOS` (`top of stack`), подверхушка -- как `NOS` (`next on stack`). ALU получает операнды из `TOS/NOS`, а результат возвращается на `Data Stack`. В Python-модели стек хранится списком, а `TOS/NOS` выводятся в журнале как derived view.

## Система команд

### Особенности ISA

- **Архитектура:** стековая.
- **Машинное слово:** одна инструкция и одна ячейка данных занимают 32 бита.
- **Адресация:** адрес указывает на 32-битное слово.
- **Доступ к памяти:** память данных доступна через `load` и `store`.
- **Поток управления:** переходы выполняются инструкциями `jmp`, `jz`, `call`, `ret`, `execute`, `iret`.
- **Ввод-вывод:** memory-mapped, специальных инструкций `in/out` нет. Устройства доступны обычными `load/store` по MMIO-адресам.

### Кодирование инструкций

Каждая инструкция занимает одно 32-битное слово.

```text
31          24 23                          0
+-------------+----------------------------+
| opcode: 8   | arg: 24                    |
+-------------+----------------------------+
```

В бинарных файлах `program.bin` и `data.bin` каждое машинное слово записывается как 4 байта в порядке **big-endian**. Внутри модели адресация остаётся пословной: адрес указывает на номер 32-битного слова. В `.hex`-листингах то же слово отображается как 8 шестнадцатеричных цифр.

### Набор инструкций

| Opcode | Мнемоника | Аргумент     | Эффект стека        | Описание                           |
| -----: | --------- | ------------ | ------------------- | ---------------------------------- |
| `0x00` | `nop`     | нет          | `--`                | пустая операция                    |
| `0x01` | `lit`     | signed imm24 | `( -- x )`          | положить литерал                   |
| `0x02` | `dup`     | нет          | `( x -- x x )`      | дублировать вершину                |
| `0x03` | `drop`    | нет          | `( x -- )`          | удалить вершину                    |
| `0x04` | `swap`    | нет          | `( a b -- b a )`    | обменять два верхних значения      |
| `0x05` | `over`    | нет          | `( a b -- a b a )`  | скопировать второе значение сверху |
| `0x10` | `add`     | нет          | `( a b -- a+b )`    | сложение                           |
| `0x11` | `sub`     | нет          | `( a b -- a-b )`    | вычитание                          |
| `0x12` | `mul`     | нет          | `( a b -- a*b )`    | умножение                          |
| `0x13` | `div`     | нет          | `( a b -- a/b )`    | целочисленное деление              |
| `0x14` | `mod`     | нет          | `( a b -- a%b )`    | остаток от деления                 |
| `0x20` | `eq`      | нет          | `( a b -- flag )`   | равно                              |
| `0x21` | `lt`      | нет          | `( a b -- flag )`   | меньше                             |
| `0x22` | `gt`      | нет          | `( a b -- flag )`   | больше                             |
| `0x30` | `load`    | нет          | `( addr -- value )` | чтение памяти данных или MMIO      |
| `0x31` | `store`   | нет          | `( value addr -- )` | запись в память данных или MMIO    |
| `0x40` | `jmp`     | addr24       | `--`                | безусловный переход                |
| `0x41` | `jz`      | addr24       | `( flag -- )`       | переход, если `flag = 0`           |
| `0x42` | `call`    | addr24       | `--`                | вызов процедуры                    |
| `0x43` | `ret`     | нет          | `--`                | возврат из процедуры               |
| `0x44` | `execute` | нет          | `( xt -- )`         | косвенный вызов процедуры          |
| `0x50` | `ei`      | нет          | `--`                | разрешить прерывания               |
| `0x51` | `di`      | нет          | `--`                | запретить прерывания               |
| `0x52` | `iret`    | нет          | `--`                | возврат из обработчика IRQ         |
| `0xFF` | `halt`    | нет          | `--`                | останов процессора                 |

### Длительности выполнения инструкций

Один вызов `Machine.step_tick()` соответствует одному переходу hardwired FSM.

| Случай                 |            Полный цикл инструкции |
| ---------------------- | --------------------------------: |
| Обычные инструкции     |                           3 такта |
| `LOAD/STORE` для MMIO  |                           3 такта |
| `LOAD` при cache hit   |                           3 такта |
| `LOAD` при cache miss  |                         13 тактов |
| `STORE` при cache hit  | 3 такта + ожидание Write Register |
| `STORE` при cache miss |                         13 тактов |
| `HALT`                 |                           2 такта |

Обычная инструкция выполняется так:

```text
FETCH -> EXECUTE -> IRQ_CHECK
```

Отдельного состояния `DECODE` нет: сырое слово инструкции декодируется комбинационно в состоянии `EXECUTE`.

| Стадия      | Тактов | Действие                                            |
| ----------- | -----: | --------------------------------------------------- |
| `FETCH`     |      1 | чтение `program_memory[PC]` в `IR`, увеличение `PC` |
| `EXECUTE`   |      1 | декодирование инструкции и её выполнение            |
| `IRQ_CHECK` |      1 | проверка прерывания между инструкциями              |

При cache miss между `EXECUTE` и `IRQ_CHECK` добавляются 10 тактов `MEM_WAIT`. Счётчик ожидания уменьшается только когда pending access получает порт памяти. `HALT` не выполняет `IRQ_CHECK`, но модель может продолжить тактировать Write Register до завершения уже принятой записи.

| `input-init`   | инициализировать программный буфер ввода        |
| `input-push`   | положить символ в буфер ввода                   |
| `input-ready?` | проверить наличие символа в буфере              |
| `input-pop`    | извлечь символ из буфера                        |
| `wait-char`    | дождаться символа в программном буфере          |

## Транслятор

Интерфейс командной строки:

```bash
python src/translator.py <source.fth> <program.bin> <data.bin> [--program-hex <program.hex>] [--data-hex <data.hex>]
```

По умолчанию создаются:

```text
program.bin      бинарная память команд
data.bin         бинарная память данных
program.bin.hex  человекочитаемый листинг команд
data.bin.hex     человекочитаемый листинг данных
```

Этапы трансляции:

1. Подключение `src/stdlib.fth` перед пользовательской программой.
2. Лексический разбор исходного кода.
3. Обработка объявлений `variable`, `buffer`, процедур `: ... ;`, обработчика `:irq ... ;` и основного кода.
4. Размещение переменных, буферов и Pascal-строк в памяти данных.
5. Генерация машинных инструкций и сохранение неразрешённых адресов.
6. Разрешение адресов процедур, переходов и `execution token` через `fixups`.
7. Запись `program.bin`, `data.bin` и человекочитаемых `.hex`-листингов.

Правила генерации кода:

* числовой литерал компилируется в `lit value`;
* встроенные арифметические слова компилируются в соответствующие ALU-инструкции;
* имя переменной или буфера компилируется как `lit address`;
* `@` компилируется в `load`, `!` — в `store`; эти же инструкции используются для MMIO;
* вызов пользовательского слова компилируется в `call address`;
* определение `: name ... ;` размещает тело процедуры в памяти команд и завершается инструкцией `ret`;
* `' name` компилируется как `lit address(name)`;
* для встроенных примитивов транслятор при необходимости создаёт скрытое слово вида `primitive; ret`, поэтому их можно использовать как `execution token`;
* `execution token` для `iret` и `halt` запрещён;
* `execute` выполняет косвенный вызов по адресу с вершины стека;
* `if/else/then` компилируются через `jz` и `jmp`, `begin/until` — через условный переход назад;
* `p"..."` размещает Pascal-строку в памяти данных и кладёт её адрес на стек;
* `."..."` размещает строку и сразу генерирует вызов `type`;
* если программа не содержит `:irq`, транслятор создаёт обработчик по умолчанию: он подтверждает IRQ через `MMIO_IRQ_ACK` и выполняет `iret`.

## Модель процессора

Интерфейс командной строки:

```bash
python src/machine.py <program.bin> <data.bin> [input.txt] [--limit <hard-limit>] [--log <log.txt>] [--output <output.txt>] [--cache | --no-cache]
```

- `--limit` задаёт защитный предел тактов; превышение является ошибкой, а не нормальной паузой;
- `--log` сохраняет полный потактовый журнал;
- `--output` сохраняет вывод программы;
- `--cache`/`--no-cache` включает или выключает восьмистрочный cache данных.

### Формат журнала

Каждая строка показывает выполненную фазу и состояние после фронта:

```text
DEBUG   machine:simulation    TICK: ... PC: ... STATE: (old->new) MODE: ... SP: ... DS: ... RS: ... IRQ: ... CACHE: ... WR: ... instr [event]
```

`STATE` показывает фактический переход FSM на этом такте. `IRQ` объединяет `E` (`irq_enable`), `P` (`irq_pending`) и `S` (`MMIO_IN_STATUS`), `CACHE` содержит накопленные hits/misses, а `WR` показывает состояние Write Register: `off`, `empty` или `busy@remaining_ticks`. В конце строки выводятся текущая инструкция и краткое событие такта.

### DataPath

![DataPath](fig/datapath.png)

Синими пунктирными линиями обозначены управляющие сигналы от Control Unit. Начальное состояние регистров задаётся при создании модели.

Основные блоки DataPath:

* `Program Memory` -- память 32-битных машинных инструкций;
* `PC` -- счётчик команд;
* `IR` -- регистр текущего 32-битного слова инструкции;
* `Sign Extend` -- знаковое расширение 24-битного аргумента инструкции `lit`;
* `Data Stack` -- стек операндов, предоставляющий значения `TOS` и `NOS`;
* `Return Stack` -- стек адресов возврата для `call`, `ret`, `execute` и прерываний;
* `ALU` -- арифметические операции и сравнения;
* `WriteBack MUX` -- выбирает значение для записи в Data Stack: литерал, результат ALU или результат чтения;
* `Next PC MUX` -- выбирает следующий адрес команды: `pc + 1`, адрес перехода из `IR`, адрес возврата, адрес обработчика IRQ или execution token;
* `Address Decoder / Access Router` -- определяет обычную память, MMIO или недопустимый адрес;
* `Data Cache / Memory Controller` -- обслуживает обращения к обычной памяти данных;
* `Data Memory` -- нижняя память данных;
* `MMIO` -- memory-mapped регистры ввода-вывода;
* `Read Data MUX` -- выбирает результат чтения из обычной памяти или MMIO;
* `Input IRQ Controller` -- хранит входной символ, флаги `ready` и `overrun` и формирует `irq_pending`.

Обычные адреса памяти данных проходят через `Data Cache`. MMIO-адреса распознаются `Address Decoder` и обходят кэш. Подтверждение входного прерывания выполняется не сигналом процессора, а записью обработчика в регистр `MMIO_IRQ_ACK`; эта запись приходит из `MMIO` в `Input IRQ Controller` как `irq_ack write`.

### Control Unit

![Control Unit](fig/control_unit.png)

Control Unit реализован как hardwired FSM. Пунктирная рамка отделяет внутреннюю логику управления от внешнего DataPath. Отдельного состояния `DECODE` нет: `opcode` из регистра `IR` комбинационно анализируется перед выполнением инструкции.

Основные блоки Control Unit:

* `Opcode Classifier` -- определяет класс инструкции: стековая операция, ALU, память, переход, IRQ или `HALT`;
* `Control Signal Generator` -- по текущему состоянию, opcode и статусным сигналам формирует управляющие сигналы для DataPath;
* `Next-State Logic` -- выбирает следующее состояние конечного автомата;
* `State Register` -- хранит текущее состояние: `FETCH`, `EXECUTE`, `MEM_WAIT`, `IRQ_CHECK` или `HALTED`;
* `IRQ Control / Flags` -- хранит флаги `IE` и `IN_IRQ`, обрабатывает команды `ei`, `di`, `iret` и формирует сигнал входа в обработчик.

## Ввод-вывод MMIO и прерывания

Ввод-вывод реализован как memory-mapped I/O. Специальных команд `in/out` нет, устройства доступны обычными `load` и `store`. Вход работает через `trap`: входной файл содержит пары `tick symbol`, например `10 Z`. Когда счётчик тактов достигает указанного значения, символ становится доступен в `MMIO_IN_DATA`, бит `ready` становится равен `1`. Если новый символ приходит при уже установленном `ready`, старый байт сохраняется, устанавливается `overrun` и увеличивается `overrun_count`

|    Адрес | Имя              | Доступ     | Назначение                                       |
| -------: | ---------------- | ---------- | ------------------------------------------------ |
| `0xFFF0` | `MMIO_IN_DATA`   | read-only  | текущий входной байт                             |
| `0xFFF1` | `MMIO_IN_STATUS` | read-only  | bit 0 `ready`, bit 1 `overrun`                   |
| `0xFFF2` | `MMIO_OUT_DATA`  | write-only | вывести младший байт значения                    |
| `0xFFF3` | `MMIO_IRQ_ACK`   | write-only | bit 0 очистить `ready`, bit 1 очистить `overrun` |

Чтение write-only регистра, запись в read-only регистр и обращение к reserved-адресу приводят к `MachineError`.

Программа разрешает прерывания командой `ei`. На стадии `IRQ_CHECK` процессор проверяет `irq_enable`, `irq_pending` и `in_irq`. При входе в обработчик текущий `PC` сохраняется в `Return Stack`, `PC` устанавливается в `IRQ_VECTOR`, `irq_enable` сбрасывается. Пользовательский обработчик `:irq` читает символ через `read-char`, подтверждает обработку через `ack-irq` и возвращается инструкцией `iret`.

## Кэш

Реализован восьмистрочный direct-mapped cache данных. Он используется только для обычной Data Memory; обращения к MMIO проходят напрямую через Address Router и cache не затрагивают.

![Data Cache](fig/cache.png)

Основные блоки схемы:

* `Address Split`  -- выделяет index и tag;
* `Cache Line Array` -- хранит строки cache;
* `Tag Comparator` и `Hit Logic` -- определяют hit или miss;
* `Memory Controller` -- управляет cache, Write Register и незавершёнными обращениями;
* `Read Result MUX` -- выбирает значение из cache или Data Memory;
* `Write Register` -- хранит одну отложенную write-through запись;
* `Pending Access` -- хранит блокирующий cache miss либо `STORE hit`, ожидающий освобождения Write Register.
* `Port Arbiter` -- распределяет единственный порт Data Memory;
* `Data Memory` -- нижняя память данных, имеет задержку 10 granted ticks.

Каждая строка cache хранит одно 32-битное слово:

```text
valid[0] | tag[12:0] | value[31:0]
```

Шестнадцатибитный адрес обычной памяти разделяется на:

```text
index[2:0] = address[2:0]
tag[12:0]  = address[15:3]
```

### Чтение

При `LOAD hit` значение из выбранной строки cache сразу передаётся через `Read Result MUX` в DataPath. Инструкция завершается в обычном состоянии `EXECUTE`.

При `LOAD miss` запрос сохраняется в `Pending Access`. После десяти тактов доступа к Data Memory полученное значение: возвращается в DataPath; записывается в выбранную строку cache; завершает инструкцию сигналом `mem_done`.

### Запись

При `STORE hit` строка cache обновляется сразу. Запись в Data Memory помещается в однозаписный `Write Register`, поэтому процессор может продолжить выполнение, не ожидая завершения write-through. Если Write Register занят, запрос временно сохраняется в `Pending Access` и ждёт освобождения регистра. После commit старой записи ожидающий STORE одновременно обновляет cache и загружает новую запись в Write Register.

При `STORE miss` процессор ожидает десять ticks. После этого значение записывается в Data Memory и одновременно размещается в cache.

### Арбитраж памяти

Data Memory имеет один порт. Приоритет доступа: `Write Register` > `Pending Access`.

Если в начале такта Write Register занят, порт получает он. Счётчик ожидающего cache miss в этот такт не уменьшается. Поэтому обращение к нижней памяти занимает десять granted ticks, но при конфликте фактическое число глобальных тактов может быть больше.
